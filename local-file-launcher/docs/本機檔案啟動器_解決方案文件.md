# 2. 背景與問題

## 2.1 現況需求

組織內有三台電腦，同仁經常需要從某個指定資料夾中找到檔案並開啟。目前以檔案總管逐層點選，缺乏一個清楚、統一的入口。期望有一個「像網頁清單一樣」的首頁，列出資料夾裡的檔案，點一下檔名就能用電腦原本的軟體（Word、Excel、PDF 閱讀器等）直接打開該檔。

## 2.2 為什麼「只做一個網頁」做不到

這是本案最關鍵的觀念。基於資訊安全，現代瀏覽器刻意限制網頁的能力：

- 網頁**不能**任意讀取你硬碟上的資料夾。
- 網頁**不能**呼叫作業系統去「用對應軟體開啟某個本機檔案」。

如果網頁可以做這些事，那麼任何一個惡意網站都能讀取、甚至執行你電腦裡的東西。因此，單純一個 HTML 檔案無法達成「列出資料夾並用原生軟體開檔」這件事。

## 2.3 解決的關鍵：本機背景小程式

要突破上述限制，必須在電腦上多一個「本機背景小程式」當作橋樑：它有權限讀取資料夾、也能請作業系統開啟檔案；瀏覽器上的網頁則透過本機網路（localhost）與它溝通。

可以用一個比喻理解：網頁就像櫃台服務員，只能在櫃台後方（瀏覽器）做事；要去倉庫（硬碟）拿東西、或啟動機器（軟體），得請一位有鑰匙的工友（本機背景小程式）代勞。本解決方案要做的，就是這位工友與它的櫃台。

# 3. 解決方案總覽

在每一台電腦上，背景執行一支小程式（以下稱「本機服務」）。它會在瀏覽器中提供一個網頁首頁，列出該台電腦指定資料夾底下的檔案；使用者點選檔名時，本機服務就用這台電腦的預設軟體開啟該檔，效果等同於在檔案總管雙擊它。

三台電腦各自安裝一份、各自指向自己的資料夾，彼此獨立、互不影響。整個服務只在本機運作，不對外開放，也不需要連上網際網路（除了取得程式更新時）。

此外，本方案會將首頁內容以「對瀏覽器擴充套件友善」的方式呈現，讓組織既有的 Chrome 擴充套件能讀取頁面內容，進行額外的運算或互動。

**核心特性一覽：**

- 本機運作、只聽本機位址，安全單純。
- 支援任意檔案類型（取決於該台電腦安裝了什麼軟體）。
- 支援子資料夾瀏覽與當前資料夾的關鍵字篩選。
- 首頁可顯示公告資訊，並可隨需求擴充其他頁面與功能。
- 提供穩定介面供既有 Chrome 擴充套件讀取與互動。
- 透過 GitHub 進行程式版本更新。
- 開機自動於背景啟動。

# 4. 使用者與使用情境

## 4.1 角色

- **一般使用者**：透過瀏覽器使用首頁，瀏覽與開啟檔案、查看公告。
- **管理者**：負責設定要顯示的資料夾、發布公告、執行版本更新。

## 4.2 使用情境（情境描述）

- 作為使用者，我打開瀏覽器書籤，就能看到指定資料夾裡的檔案清單。
- 作為使用者，我點一個檔名，它就用 Word／Excel／PDF 閱讀器等原生軟體開啟。
- 作為使用者，我可以點進子資料夾，或用篩選框快速找到檔案。
- 作為使用者，我在首頁能看到組織最新公告。
- 作為管理者，我可以修改設定，指定這台電腦要顯示哪個資料夾。
- 作為管理者，我可以用一個更新動作，把程式更新到最新版本。
- 作為既有擴充套件，當使用者瀏覽清單時，我能讀取頁面內容並進行額外運算或互動。

# 5. 系統架構

## 5.1 組成元件

| 元件 | 說明 | 所在位置 |
|------|------|----------|
| 本機服務 | 以 Python（Flask）撰寫，提供首頁、列檔、開檔功能 | 每台電腦背景執行 |
| 前端首頁 | 由本機服務提供的網頁（HTML／CSS／JS） | 本機 Chrome 瀏覽器 |
| Chrome 擴充套件 | 組織既有、獨立維護的元件，讀取首頁內容 | 本機 Chrome 瀏覽器 |
| 指定資料夾 | 檔案的實際存放處 | 電腦檔案系統 |
| 原生軟體 | 最終開啟檔案的應用程式（Word、Excel…） | 電腦作業系統 |

## 5.2 架構示意

下圖以層次表示各元件的所在位置與連線關係（正式架構圖為後續產出之獨立產物）。

```
【本機 Chrome 瀏覽器】
   ├─ 前端首頁（HTML / CSS / JS）
   └─ Chrome 擴充套件（content script）
        ↑↓  讀取頁面內容 / 雙向互動（與前端首頁）
        │
        │   HTTP（127.0.0.1）
        ▼
【本機服務（Python＋Flask）】　提供頁面 · 列出清單 · 開啟檔案
        │
        ├─ 讀取清單 ──▶ 〔指定資料夾（檔案來源）〕
        └─ 開啟檔案 ──▶ 〔原生軟體：Word / Excel / PDF〕
```

## 5.3 資料流

1. 使用者用瀏覽器開啟 `http://localhost:埠號`，本機服務回傳首頁。
2. 首頁向本機服務索取清單；本機服務讀取指定資料夾，回傳檔案資料，首頁渲染成連結。
3. 使用者點選某個檔名；首頁送出開檔請求；本機服務以預設軟體開啟對應檔案。
4. Chrome 擴充套件在旁讀取首頁內容，進行額外運算或互動。

## 5.4 重要邊界與原則

- 本機服務只監聽本機位址（127.0.0.1），同網段其他電腦與外部皆無法存取。
- 前端首頁與擴充套件都在瀏覽器沙箱內，**兩者都無法自行開啟本機軟體**；「真正開檔」一律由本機服務執行。
- 三台電腦各自獨立，不共享資料、不互相連線。

# 7. 與 Chrome 擴充套件的整合

## 7.1 定位

組織既有的 Chrome 擴充套件被視為「獨立維護的既有元件」。本解決方案不規範它內部做什麼運算，只負責提供一份**穩定、有文件的介面**，讓它能可靠地讀取首頁內容並（必要時）觸發動作。

## 7.2 整合方式（概念）

- **讀取頁面內容**：擴充套件以 content script 注入首頁後，可直接讀取頁面的檔案清單。為了讓它讀得乾淨，首頁會把每個檔案項目以語意化連結與資料屬性輸出，並另外提供一份內嵌的結構化資料。
- **即時通知**：每次清單載入完成，首頁會發出事件通知，擴充套件可即時反應，不必反覆查詢。
- **反向互動**：首頁同時開放一個訊息入口，讓擴充套件可請首頁執行「開啟某檔」或「重新整理」等動作。

介面的精確規格（資料屬性、事件名稱、訊息格式）見第 14 章附錄 E、F。

# 9. 技術選型

## 9.1 後端：Python ＋ Flask（確定採用）

本案後端確定採用 Python 搭配 Flask 框架，理由如下：

- **模板引擎（Jinja2）支援版型繼承**：可建立共用的版型（導覽列、頁首頁尾），新增頁面（如公告頁）只需新增一條路由與一個模板，是「方便後續頁面開發」的關鍵。
- **模組化（Blueprints）**：可將「檔案清單」「公告」「其他功能」切分為獨立模組，程式成長時結構清晰。
- **輕量、社群成熟**：學習曲線平緩、範例與文件豐富。
- **與既有 Python 取向一致**：便於團隊維護。

## 9.2 為何不選其他方案

- **FastAPI**：強項在 API 與非同步，適合「後端只出 JSON、前端為獨立單頁應用」的取向；本案以伺服器端渲染頁面為主，採 Flask 更順手。
- **Django**：功能全包但偏重，適合需要資料庫、後台管理、帳號權限的大型應用；對本案屬過度設計。
- **Node／Nuxt 等 JS 方案**：可行，但部署需 Node 環境與建置步驟，較本案的輕量取向為重；除非團隊全面採用 JS 生態，否則不建議。

## 9.3 前端

採伺服器端渲染的 HTML（Jinja 模板）搭配原生 CSS／JS，不使用重型前端框架。此做法部署簡單、可離線運作，且內容於初始 HTML 即完整呈現，對擴充套件讀取最為友善。

## 9.4 開檔機制

於 Windows 以系統預設關聯程式開啟檔案（等同雙擊行為）。

# 10. 部署與維運

## 10.1 系統需求（每台電腦）

- Windows 10 或 11
- Python 3（安裝時勾選 Add Python to PATH）
- Git

## 10.2 安裝步驟（每台電腦一次）

1. 安裝 Python 與 Git。
2. 以 Git 取得程式（`git clone`）。
3. 安裝相依套件（`pip install -r requirements.txt`）。
4. 複製設定範本為正式設定檔，並填入這台電腦要顯示的資料夾路徑。
5. 測試啟動，確認瀏覽器能看到清單並可開檔。
6. 設定開機自動於背景啟動。

## 10.3 常駐執行

服務以背景方式常駐執行（不顯示主控台視窗），並設定為開機自動啟動。使用者平時以瀏覽器書籤連至首頁即可。

## 10.4 版本更新（透過 GitHub）

當有新版本時，於程式資料夾執行更新動作（`git pull`，必要時再執行一次 `pip install`），重新啟動服務即生效。各台電腦的設定檔不納入版本控管，更新不會覆蓋各機設定。

## 10.5 公告維護

公告內容由管理者以簡單的資料檔（見附錄）維護，更新後即顯示於首頁，毋須改動程式。

## 10.6 疑難排解（摘要）

- 網頁顯示「連不上本機服務」：服務未啟動，重新啟動背景程式。
- 點檔案沒反應：該電腦未安裝可開啟該類型的軟體。
- 找不到 python：安裝 Python 時未勾選加入 PATH，重裝並勾選。
- 埠號被占用：服務會自動改用下一個可用埠，或於設定檔調整。

# 11. 使用追蹤與集中彙整

## 11.1 目的

記錄使用者在工具上的行為（瀏覽了哪些資料夾與頁面、開啟了哪個檔案），並將三台電腦的紀錄彙整到一處，供管理者了解使用情形，例如最常被開啟的檔案、各台電腦的使用活躍度。

## 11.2 運作方式（共用網路資料夾彙整）

由於組織沒有常時開機的伺服器、但有共用網路資料夾，故採「共用資料夾彙整」：

- 每台電腦把自己的使用事件寫成一個專屬的紀錄檔，放到共用網路資料夾（例如 `PC-A.jsonl`、`PC-B.jsonl`、`PC-C.jsonl`）。
- 各台只寫自己的紀錄檔，互不干擾，避免多台同時寫入同一檔造成衝突。
- 管理者要查看時，由報表功能把三份紀錄讀進來合併，產生統計。

```
PC-A 啟動器 ─┐
PC-B 啟動器 ─┼─▶ 共用網路資料夾　◀─ 管理者報表頁（需要時讀取並合併）
PC-C 啟動器 ─┘     PC-A.jsonl／PC-B.jsonl／PC-C.jsonl
（各台先寫本機暫存，再補送自己的紀錄檔；連不到共用資料夾時，暫存稍後自動補送）
```

## 11.3 容錯（重要）

追蹤是非阻斷且容錯的：事件先寫進本機暫存，再由背景程序補送到共用資料夾。萬一當下連不到共用資料夾，事件會留在本機、稍後自動補送；檔案啟動器本身照常運作，絕不會因為追蹤而受影響。

## 11.4 紀錄內容

每筆事件包含：機器名稱、Windows 使用者、時間、事件類型（看頁面／進資料夾／開檔）、對應路徑。

## 11.5 查看報表

由於沒有常開主機，報表採「需要時查看」的方式：管理者於自己的電腦開啟管理頁面，由它即時讀取共用資料夾的三份紀錄、合併後以圖表呈現（最常開啟的檔案、各機與各使用者的活動、時間趨勢）。

## 11.6 隱私與保存

- 事先向使用者公開告知追蹤的用途與範圍。
- 設定保存期限（例如僅保留 N 天），逾期自動清理；各台只清理自己的紀錄檔。
- 僅記錄必要欄位；管理報表僅限管理者存取，共用資料夾權限亦應限制。

## 11.7 POC 延伸：使用者統計、生產力分析與正式化考量

概念驗證（POC）階段為展示能力，於追蹤之上額外實作了下列功能：

- **登入與使用者識別**：POC 改以「使用者自行輸入名稱、存入 Cookie（約 30 天）」作為識別，取代原規格「以 Windows 帳號（`getpass.getuser()`）自動辨識」的做法。無密碼驗證，僅供內網識別。
- **使用者統計分析頁**：於整體報表（最常開啟檔案、各維度、每日趨勢）之外，另提供個別使用者的詳細分析——每日趨勢、熱門檔案 TOP 10、24 小時時段分析、檔案類型分布、週統計摘要。
- **生產力分數與排行榜**：以開檔次數推估「節省時間」與「生產力分數」，並提供使用者排行榜。

**正式化時要決定／注意的事項：**

- 生產力分數與排行榜一旦進入正式環境，性質即從「使用統計」轉為「員工績效評估」，超出本文件原訂範圍，亦屬先前列為範圍外的使用者管控領域。正式採用前須有明確的告知與政策，並重新檢視指標合理性（「開檔次數＝生產力」「每次省 30 秒」為展示性假設，非經驗證之生產力衡量）。
- 自填名稱無驗證，任何人可輸入任意名稱，統計歸戶不可信。若正式版的任何決策需依賴這些統計，使用者識別應改回可信來源（如 Windows 帳號）。
- POC 報表中以「各瀏覽器」為一個統計維度，與設定中的 `machine_name`（機器維度）可能不一致；正式化前需與開發確認該維度的實際依據（User-Agent 或機器）。

## 14.C API 端點規格

### GET `/`
回傳首頁 HTML（含檔案清單與公告區塊）。

### GET `/api/list?rel=<相對路徑>`
列出 `folder` 底下、相對路徑 `rel` 所指資料夾的內容。`rel` 省略代表根目錄。

回應（成功）：

```json
{
  "ok": true,
  "rel": "報表",
  "parent": "",
  "is_root": false,
  "crumbs": [{ "name": "報表", "rel": "報表" }],
  "entries": [
    { "name": "2024", "type": "dir",  "size": 0,     "mtime": "2026-06-18 09:12", "ext": "" },
    { "name": "月報.xlsx", "type": "file", "size": 38912, "mtime": "2026-06-19 08:55", "ext": "xlsx" }
  ]
}
```

回應（失敗）：`{ "ok": false, "error": "說明文字" }`，並回傳對應的 HTTP 狀態碼（400／500）。

### POST `/api/open`
要求以預設軟體開啟某檔。請求本文：`{ "rel": "報表/月報.xlsx" }`。

- 成功：`{ "ok": true, "name": "月報.xlsx" }`
- 失敗：`{ "ok": false, "error": "說明文字" }`

### GET `/api/announcements`
回傳公告清單：

```json
{ "ok": true, "items": [
  { "title": "系統維護通知", "date": "2026-06-15", "body": "本週五進行維護。" }
] }
```

### 路徑安全規則
所有帶 `rel` 的請求，後端必須：將 `folder` 正規化為基準路徑；把 `rel` 接上後正規化為目標路徑；確認目標路徑等於基準路徑或位於其下（前綴比對），否則拒絕。藉此防止 `..` 等路徑穿越。

## 14.E 前端「擴充套件友善」輸出規格

**每個檔案項目以連結與資料屬性輸出：**

```html
<a class="file-link"
   href="#"
   data-rel="報表/月報.xlsx"
   data-name="月報.xlsx"
   data-type="file"
   data-ext="xlsx"
   data-size="38912"
   data-mtime="2026-06-19 08:55">月報.xlsx</a>
```

資料夾則以 `class="dir-link"` 與 `data-type="dir"` 輸出。

**內嵌結構化資料（供直接解析）：**

```html
<script id="filelist-data" type="application/json">
{ "rel": "報表", "entries": [ /* 同 /api/list 的 entries */ ] }
</script>
```

**清單載入完成時，於頁面發出通知：**

- 自訂事件：在 `document` 上 `dispatchEvent` 一個 `filelauncher:list`，其 `detail` 為 `{ rel, entries }`。
- 視窗訊息：`window.postMessage({ source: "file-launcher", type: "list", payload: { rel, entries } }, "*")`。

## 14.F 擴充套件整合協定（反向互動）

首頁監聽 `window` 的 `message` 事件，接受來源為擴充套件的請求：


## 14.I 使用追蹤與集中彙整（技術規格）

**設定（`config.json` 內的 `tracking` 區段）**

| 欄位 | 型別 | 說明 |
|------|------|------|
| `enabled` | 布林 | 是否啟用追蹤。 |
| `machine_name` | 字串 | 此機器於報表顯示的名稱（預設為主機名稱）。 |
| `shared_log_dir` | 字串 | 共用網路資料夾路徑（UNC，如 `\\server\share\file-launcher-logs`）。 |
| `flush_interval_seconds` | 整數 | 由本機暫存補送至共用資料夾的間隔（秒）。 |
| `retention_days` | 整數 | 紀錄保存天數，逾期自動清理。 |

**事件紀錄檔（每台一檔，JSON Lines）**

- 檔名：`<machine_name>.jsonl`，置於 `shared_log_dir`。
- 每行一筆 JSON 事件，例如：

```json
{"ts":"2026-06-19T08:55:12+08:00","machine":"PC-A","user":"alice","type":"open","path":"報表/月報.xlsx"}
```

- `type` 取值：`view`（頁面）／`list`（資料夾）／`open`（檔案）。

**本機暫存與補送（容錯）**

- 每台以本機 SQLite（`usage.sqlite3`，含 `synced` 旗標）作為持久暫存：使用者操作時先寫入暫存，必定成功且不阻斷。
- 背景程序每 `flush_interval_seconds` 將未同步事件附加到該機自己的 `<machine_name>.jsonl`，成功後標記為已同步。
- 共用資料夾不可達時，事件留在暫存、下次重試；核心功能不受影響。
- 各台僅附加與清理自己的紀錄檔，避免跨機寫入衝突。

**彙整與報表**

- 管理頁（限管理者）即時讀取 `shared_log_dir` 下所有 `*.jsonl`，逐行解析合併，產生統計（最常開啟檔案、各機與各使用者活動、時間分佈），以 Chart.js 呈現。
- 亦可提供獨立報表腳本，產生靜態 HTML 報表。

**保存與清理**

- 各台依 `retention_days` 清理自己的 `<machine_name>.jsonl` 與本機暫存中過期事件（重寫自己的檔；因僅該機擁有，無衝突）。

**安全與隱私**

- 共用資料夾權限限定相關人員；管理報表頁僅管理者可存取。
- 僅儲存必要欄位；對使用者公開告知；設定明確保存期限。

## 14.J POC 延伸功能（技術規格）

> 下列為 POC 已實作、超出原核心規格的部分；正式版是否保留另議（見 11.7）。

**登入（Cookie-based）**

- 首次造訪導向登入頁；輸入名稱即建立 Cookie（效期約 30 天），可登出清除。
- Cookie 採 `HttpOnly` 與 `SameSite=Lax`；無密碼驗證（內網識別用途）。
- 路由（依 POC）：`GET /login`、`POST /login`、`GET /logout`（或等義）。
- 追蹤事件的 `user` 於 POC 取自此 Cookie 名稱，而非 `getpass.getuser()`。

**使用者統計分析**

- 頁面：`GET /admin/user-stats`，搭配對應資料端點回傳個別使用者彙整。
- 指標：總開檔次數、總瀏覽次數、節省時間、生產力分數。
- 視覺化：每日趨勢、熱門檔案 TOP 10、24 小時時段、檔案類型分布、週統計。
- 支援期間切換（本週／本月）與使用者排行榜（依生產力分數）。

**生產力計算（POC 展示性公式，非經驗證）**

```
節省時間 = 開檔次數 × 30 秒
生產力分數 = min(100, 開檔次數 × 2.5)
```

**整體報表維度（待確認）**

- POC 報表列出「各瀏覽器使用統計」，與設定 `machine_name`（機器維度）可能不一致；需與開發確認實際統計依據。

**擴充套件快捷鍵（POC 新增）**

- `Alt+R` 重新整理、`Alt+H` 回根目錄。
